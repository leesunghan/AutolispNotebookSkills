# 완료 정의 (Definition of Done, DoD) — AutoLISP 노트북 표준

이 프로젝트의 완료 정의는 AutoCAD/BricsCAD AutoLISP 노트북(`.lspnb`)의 실행 안전성, 재현성 및 무결성을 보장하기 위해 작성되었습니다. 모든 작업(태스크)은 다음 품질 기준을 모두 충족해야 완료된 것으로 간주합니다.

---

## 1. 파일 구조 및 노트북 표준
- [ ] **번갈아 배치되는 셀 구조**: 모든 노트북 파일은 `마크다운 설명 셀 ↔ 실행 가능한 AutoLISP 코드 셀`이 교대로 배치되되, 리습 함수 및 단계별로 설명 마크다운과 코드 셀을 촘촘하게 세분화하여 구성해야 합니다.
- [ ] **Progn 배제 및 셀 세분화**: 개별 코드 셀에 불필요한 `(progn ...)` 래핑을 배제하여 코드 투명성을 유지합니다.
- [ ] **마크다운 3단계 Depth**: 노트북 내의 마크다운 설명 셀은 3단계 depth 헤더 구조(#, ##, ###)를 엄격히 준수하여 학습적 가독성을 제공해야 합니다.
- [ ] **JSON 유효성**: `.lspnb` 노트북 파일이 올바른 JSON 구조로 저장되고 VS Code Notebook 확장 기능에서 구문 오류 없이 로드되는지 확인합니다.

---

## 2. 변수 범위 선언 및 네임스페이스 안전성
- [ ] **변수 지역화**: 함수 내의 모든 임시 변수는 `(defun c:function-name (/ local1 local2 ...))`와 같이 선언부에 반드시 지역 변수로 등록해야 합니다. 전역 변수가 임의로 생성되어 캐시 또는 CAD 네임스페이스를 오염시켜서는 안 됩니다.
- [ ] **유일성 접두사**: 헬퍼 함수 및 고유 유틸리티는 다른 타사 플러그인과의 이름 충돌을 방지하기 위해 유일한 접두사(예: `myproject:`)를 사용해야 합니다.

---

## 3. CAD 환경 및 시스템 변수 보존
- [ ] **환경 복원**: 수정이 필요한 CAD 시스템 변수(`OSMODE`, `CMDECHO`, `NOMUTT`, `ANGBASE` 등)는 코드 실행 전 원래 값을 백업하고, 정상 종료 시점 또는 에러가 발생한 시점에 반드시 복원해야 합니다.
- [ ] **로컬 에러 핸들러**: 실행 취소(Esc)나 예외 발생 시 캐싱해 둔 시스템 변수를 온전히 복원하기 위해 함수 수준의 로컬 `*error*` 핸들러가 반드시 선언되어야 합니다.
- [ ] **Undo 경계**: 세분화된 여러 코드 셀들의 도면 변경 사항들을 연계된 실행 취소 그룹화(`_.undo _group` / `_.undo _end` 또는 `vla-StartUndoMark` / `vla-EndUndoMark`)로 묶어 단 한 번의 실행 취소(`Ctrl+Z`)로 복구되도록 처리해야 합니다.

---

## 4. 검증 및 테스트
- [ ] **데이터베이스 어설션**: 기능 개발 시 반드시 CAD 도면 데이터베이스 상태를 점검하는 검증 셀을 작성해야 합니다. (`tblsearch`를 통한 레이어/블록 존재 여부 확인, DXF 필터가 포함된 `ssget`으로 도면 객체 수 검증 등)
- [ ] **TDD / Prove-It 워크플로우**: 모든 신규 기능 또는 버그 수정은 실패하는 어설션(RED 단계)을 먼저 작성하고, 이를 해결하는 제도 로직(GREEN 단계)을 작성한 후 구조를 개선(REFACTOR 단계)하는 TDD 사이클을 거쳐야 합니다.
- [ ] **좌표 독립성**: 도면의 기하 연산은 현재 활성화된 사용자 좌표계(UCS)나 뷰포트 배율에 영향을 받지 않도록 좌표 변환을 완벽히 계산해야 합니다.

---

## 5. 성능 및 성능 감사
- [ ] **ActiveX API 최적화**: 대량의 도면 객체를 다루는 반복 루프에서는 느린 커맨드 프롬프트 `(command ...)` 방식 대신 ActiveX VLA 함수(`vla-AddLine`, `vla-AddLightWeightPolyline` 등)를 사용하여 실행 속도를 최적화해야 합니다.
- [ ] **선택 세트 메모리 해제**: 임시로 할당한 selection set 변수는 사용 후 `nil`을 할당하여 메모리 누수를 방지합니다.

---

## 6. 검토 및 배포
- [ ] **리뷰 및 배포 승인**: `/ship` 오케스트레이션 명령을 통해 코드 리뷰, 안전 감사, 테스트 커버리지 보고서를 병합하여 배포 승인(GO verdict)을 얻고 롤백 계획을 명시합니다.
