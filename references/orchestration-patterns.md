# 에이전트 오케스트레이션 패턴 (CAD 자동화 에디션)

이 저장소에서 지원하는 에이전트 오케스트레이션 패턴 및 방지해야 할 안티패턴에 관한 지침입니다. 여러 페르소나를 조율하는 새로운 슬래시 명령어(slash command)를 추가하거나 기존 페르소나를 감싸는 새로운 역할을 도입하기 전에 이 가이드를 읽어보십시오.

핵심 규칙: **오케스트레이터 역할은 오직 사용자(또는 슬래시 명령어)입니다. 각 페르소나가 또 다른 페르소나를 임의로 호출하지 않습니다.** 스킬은 각 페르소나의 워크플로우에 정의된 필수 수행 절차입니다.

---

## 공식 권장 패턴

### 1. 직접 호출 (Orchestration 없음)

단일 페르소나가 단일 관점에서 하나의 리포트만 생산하여 반환하는 패턴입니다. 가장 기본적이고 비용 효율적입니다.

```
사용자 → code-reviewer → 리포트 → 사용자
```

**사용 시점:** 작업 내용이 하나의 파일이나 노트북에 관한 단일 관점이고, 한 문장으로 설명할 수 있는 수준일 때 사용합니다.

**적용 예시:**
- "이 노트북의 지역 변수 선언 상태를 검토해줘" → `code-reviewer` 호출
- "이 `door-generator.lsp` 코드의 OSNAP 설정 및 루프 안정성을 검사해줘" → `security-auditor` 호출
- "이 치수선 튜토리얼에 누락된 CAD DB 어설션 셀이 무엇인지 알려줘" → `test-engineer` 호출

**비용**: 단 1회 왕복.

---

### 2. 단일 페르소나 슬래시 명령어

하나의 페르소나에 프로젝트의 전용 스킬을 결합하여 단축 명령어로 구성한 것입니다. 사용자가 매번 번거롭게 작업을 수동으로 지시할 필요를 덜어줍니다.

```
/review → code-reviewer (code-review-and-quality 스킬 수행) → 리뷰 리포트 출력
```

**사용 시점:** 동일한 세팅을 가진 단일 페르소나 실행을 반복적으로 간편하게 트리거하고자 할 때 사용합니다.

**이 저장소의 적용 사례:** `/review`, `/test`, `/code-simplify`.

---

### 3. 병렬 팬아웃 및 병합 (Parallel Fan-Out with Merge)

여러 에이전트 페르소나가 동일한 코드나 노트북을 동시(병렬)에 읽어 각자 독립된 보고서를 생성합니다. 이후 메인 에이전트가 이를 하나로 수집 및 가공(Merge)하여 종합 보고서와 최종 의사결정을 도출합니다.

```
                    ┌─→ code-reviewer    ─┐
/ship → 팬아웃  ───┼─→ security-auditor ─┤→ 병합(Merge) → 최종 배포 여부 및 롤백 안
                    └─→ test-engineer    ─┘
```

**사용 시점:**
- 수행할 하위 작업들이 서로 완벽히 독립적일 때 (상호 간의 의존성이나 실행 순서가 상관없음).
- 하위 에이전트들이 각자의 컨텍스트 창(context window)을 활용하여 깊이 있게 검증해야 할 때.
- 메인 에이전트의 컨텍스트 창이 취합된 리포트들을 처리하기에 충분할 때.
- 전체적인 작업 응답 속도(latency)가 중요할 때.

**이 저장소의 적용 사례:** `/ship`.

---

### 4. 사용자 주도 순차적 파이프라인 (Sequential Pipeline as User-Driven Slash Commands)

사용자가 상황 판단에 따라 정의된 단계 순서대로 슬래시 명령어를 입력하며 컨텍스트를 계승해 나가는 방식입니다. 이를 조율하는 자동 오케스트레이터 에이전트는 없으며, 사용자가 스스로 조율자가 됩니다.

```
사용자 실행 순서:  /spec  →  /plan  →  /build  →  /test  →  /review  →  /ship
```

**사용 시점:** 작업 단계별로 명확한 의존성(이전 단계의 결과물이 다음 단계의 입력값이 됨)이 존재하며, 매 단계 사이에 사람이 직접 보고 개입/판단하는 것이 더 합리적일 때 사용합니다.

**이 저장소의 적용 사례:** AutoLISP Notebook 생명주기 전체 단계 (DEFINE → PLAN → BUILD → VERIFY → REVIEW → SHIP).

---

### 5. 리서치 격리 (Research Isolation / Context Preservation)

참고 도서나 대용량의 CAD 개발자 문서(API 매뉴얼 등)를 훑어봐야 하는 등 컨텍스트가 너무 큰 리서치 작업을 처리할 때, 메인 세션 창을 어지럽히지 않기 위해 연구(Research) 전용 하위 에이전트를 독립적으로 작동시켜 핵심 요약본만 넘겨받는 패턴입니다.

```
메인 에이전트 → 리서치 하위 에이전트 (API 문서 20개 탐색) → 핵심 요약 전달 → 메인 작업 계속 수행
```

**사용 시점:**
- 메인 세션이 후속 구현 작업에만 온전히 집중해야 할 때.
- 방대한 참고 데이터 대비 추출되는 요약본의 용량이 매우 작을 때.

## Claude Code 호환성

이 가이드는 범용적인 원리를 설명하지만, 대부분의 사용자 환경인 Claude Code CLI의 연동 방식을 전제로 합니다.

### 페르소나 위치
하위 에이전트 페르소나 정의 파일은 저장소 루트의 `agents/` 폴더에 마크다운 파일로 위치합니다. 예를 들어 `agents/code-reviewer.md`, `agents/security-auditor.md`, `agents/test-engineer.md` 등으로 작성되면 CLI 환경에서 자동으로 인식됩니다.

### 하위 에이전트(Subagents) vs 에이전트 팀(Agent Teams)
Claude Code 환경에서는 병렬 조율을 위해 **Subagents**(메인 에이전트가 각 하위 에이전트에 지시하고 취합만 함)와 **Agent Teams**(에이전트들이 상호 대화하며 작업을 공동 분담) 방식을 지원합니다.

| 분류 | Subagents (하위 에이전트) | Agent Teams (에이전트 팀) |
|---|---|---|
| 조율 방식 | 메인이 지시하고 결과 보고서만 받음 | 서로 메시지를 주고받으며 작업 목록을 공유 |
| 컨텍스트 관리 | 각 에이전트별 독립된 컨텍스트 공간 유지 | 각 에이전트별 독립된 컨텍스트 공간 유지 |
| 추천 용도 | 독립적으로 작성된 결과물들의 단순 취합 및 리포트화 | 하나의 복잡한 문제 해결을 위해 상호 토론과 협업이 필요할 때 |
| 비용 | 비교적 저렴함 | 대화가 누적되므로 비용이 높음 |

---

## 작동 사례: 교차 디버깅을 위한 에이전트 팀 구성

병렬로 단순 취합하는 `/ship` 명령어와 달리, 상호 협조가 필요한 **에이전트 팀**을 언제 사용해야 하는지에 관한 대표 사례입니다.

### 시나리오

> *벽체 그리기 LISP 루틴을 돌릴 때 가끔씩 벽체 선이 500mm 정도 엉뚱하게 밀려서 그려지는 버그가 있습니다. 약 50번 실행할 때 1번꼴로 간헐적으로 나타납니다. 에러 메시지는 출력되지 않으며, 지난주 코드 최적화 이후 발생하기 시작했습니다.*

고려할 수 있는 몇 가지 예상 가설들:

1. **OSNAP 간섭**: 벽체를 그리는 `command` 함수가 구동될 때 화면 근처의 엉뚱한 점(끝점이나 중간점 등)에 스냅이 걸려 선 좌표가 밀려버림.
2. **UCS 좌표계 회전**: 사용자의 작업 화면이 현재 WCS가 아닌 회전된 UCS 상태인데, LISP 코드 상에서 WCS 기준으로 좌표값을 집어넣어 발생함.
3. **전역 네임스페이스 오염**: 루프 내부의 인덱스 변수를 지역 변수로 등록하지 않아, 이전에 돌렸던 다른 셀의 LISP 전역 변수 찌꺼기 값을 물려받아 연산됨.
4. **소수점 반올림 오류**: 좌표를 동등 비교할 때 LISP의 단순 `=` 연산자를 사용하여 정밀도 미세 편차로 인해 가끔 조건문이 어긋남.

단일 에이전트는 하나의 가설만 꼽은 후 조사를 멈출 것입니다. 단순히 각자의 결과만 리포트하는 `/ship` 형태의 subagent 팬아웃 역시 서로의 리포트 내용을 모르기 때문에 적절한 대안을 토론하여 교차 검증하지 못합니다.

이때 **에이전트 팀**을 꾸려 대응합니다:
- `code-reviewer`는 루프 스코프 내부의 전역 변수 유출 가능성을 정밀 추적합니다.
- `security-auditor`는 현재 도면 상태와 뷰포트의 시스템 변수(UCS, OSMODE, ANGBASE) 설정을 분석합니다.
- `test-engineer`는 OSNAP이 켜진 상태와 꺼진 상태에서 각각 해당 오차가 일어나는지 시험하는 테스트 코드를 작성합니다.

이들은 메시지를 주고받으며 각자의 탐색 성과를 공유하고 서로의 이론을 검증한 뒤, 최종적으로 "OSNAP 간섭"이 원인이었음을 밝혀내고 이를 완화할 테스트 통과 셀을 만들어 냅니다.

## 방지해야 할 안티패턴

### A. 라우터 페르소나 (Router Persona)
다음에 어떤 페르소나를 호출할지 결정하는 업무만 맡는 메타 오케스트레이터 에이전트입니다. 컨텍스트 낭비와 불필요한 토큰 비용만 초래하므로 슬래시 명령어로 직접 분기하십시오.

### B. 페르소나가 또 다른 페르소나를 호출하는 구조
리뷰 에이전트인 `code-reviewer`가 검사 도중 안전성에 문제가 있어 보인다고 스스로 `security-auditor`를 불러 추가 검사를 가하는 형태입니다. 대신 코드 리뷰 리포트에 "보안/안전성에 우려가 있으니 추가 안전성 감사를 권고함"과 같이 적는 것이 올바릅니다.

### C. 사용자를 배제한 연속 실행 조율자
spec을 부르고 자동으로 plan을 돌린 다음 build까지 혼자 연쇄적으로 처리하는 자동 조율자 방식입니다. 이는 사용자의 승인 체크포인트를 제거하여 에이전트가 예기치 않게 경로를 이탈(drift)하게 만듭니다. 순서 지시는 항상 사용자가 하도록 주도권을 넘기십시오.
