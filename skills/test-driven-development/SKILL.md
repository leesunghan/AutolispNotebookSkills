# 테스트 주도 개발 (AutoLISP Notebook 에디션)

## 개요

CAD 자동화 영역에서 육안으로 화면을 대충 보고 "동작하는 것 같다"며 넘어가는 것은 설계 검증이 아닙니다. 실제 그리기 명령을 실행해 도면에 선을 치기 전, 목표 도면 상태(레이어 존재 여부, 예상 개체 좌표, 캐드 상태 변수 등)를 질의하여 통과 여부를 검출해내는 데이터베이스 검증 및 어설션(assertion) 코드를 먼저 작성하십시오. 이를 기반으로 기능 구현에 나서야 안전한 개발이 가능합니다.

## CAD TDD 사이클

```
     RED (실패)              GREEN (성공)            REFACTOR (리팩토링)
검증 어설션 코드          최소한의 AutoLISP 그리기        AutoLISP 중첩 구조 정리
셀을 작성해 실패 확인 ──→ 코드를 작성해 통과 유도 ──→ 및 변수 로컬화 수행 ──→ (순환 반복)
     │                       │                        │
     ▼                       ▼                        ▼
  검증 실패                 검증 통과                검증 여전히 통과
```

### Step 1: RED — 실패하는 어설션(Assertion) 셀 작성
캐드 데이터베이스 테이블을 파싱하여 목표 도면층, 선종류, 혹은 폴리선 객체의 기하 좌표 정보가 누락되어 검출되지 않는 상태를 판별하는 실패 유도 어설션 셀을 작성합니다.

```lisp
;; RED: 목표 구조 벽체(S-WALL-CONC) 도면층이나 형상 폴리선이 도면에 없으므로 실패를 반환합니다.
(progn
  (setq passed T)
  (if (not (tblsearch "LAYER" "S-WALL-CONC"))
      (progn (princ "\n[FAIL] S-WALL-CONC 도면층을 찾을 수 없습니다.") (setq passed nil))
  )
  (if (not (ssget "_X" '((0 . "LWPOLYLINE") (8 . "S-WALL-CONC"))))
      (progn (princ "\n[FAIL] 벽체 폴리선 엔티티를 찾을 수 없습니다.") (setq passed nil))
  )
  (if passed (princ "\n[PASS] 벽체 생성 및 레이어가 정상 입증되었습니다!"))
  (princ)
)
```

### Step 2: GREEN — 통과하기 위한 최소한의 작도 코드 구현
이전 단계의 검증 코드가 성공적으로 돌 수 있도록 레이어를 개설하고 선을 그어주는 핵심 AutoLISP 구문을 구현합니다.

```lisp
;; GREEN: 레이어 생성 명령을 실행하고 사각형 폴리선을 제도합니다.
(progn
  (command "._layer" "_make" "S-WALL-CONC" "_color" "3" "" "")
  (command "._pline" "0,0" "2000,0" "2000,2000" "0,2000" "_close")
  (princ "\n임시 폴리선 그리기 완료.")
  (princ)
)
```

### Step 3: REFACTOR — 구조 개선 및 다듬기
검증 셀이 통과(Green)되는 것을 전제로 AutoLISP 소스의 구조를 정돈합니다:
- 모든 임시 변수를 매개변수 리스트로 숨겨 로컬화(local variable declaration)를 완수합니다.
- 콘솔 명령어 방식 그리기 대신 속도가 빠른 ActiveX 방식으로 점진 치환합니다.
- 수정한 뒤 검증 어설션 셀을 재실행하여 동작 무결성이 깨지지 않았는지 교차 확인합니다.

## CAD 버그 검증 패턴 (The CAD Prove-It Pattern)

버그 피드백(예: "모서리 모따기 기능 수행 시, 예각 상태의 코너 부분선들이 서로 겹치거나 제대로 트림되지 않고 남아있습니다")이 접수되면:
1. **버그 재현 검증 셀 작성**: 예상되는 정상 연결 좌표값을 질의하는 AutoLISP 코드를 적어 실행시킵니다. 실제로 겹친 채 방치되어 있으므로 어설션이 실패(RED)함을 눈으로 확인합니다.
2. **해결책 코딩**: 모서리 벡터의 각도가 일정 기준치 이하일 때 예외 분기 처리하는 모따기 정리 코드를 얹습니다.
3. **재시도**: 검증 셀을 다시 돌려 어설션이 정상(PASS)으로 돌아왔는지 확인합니다.

## CAD 데이터베이스 어설션(Assertion) 유형

텍스트 결과 리턴값뿐만 아니라, 캐드 자체 도면층 및 엔티티 데이터 테이블을 다차원적으로 감사하십시오.

| 감사 레벨 | 검증 AutoLISP 코드 예시 | 검증하는 핵심 항목 |
|------------|-------------------------|------------------|
| **Layer Table (도면층 테이블)** | `(tblsearch "LAYER" name)` | 특정 표준 도면층이 정확한 인덱스 색상과 세팅으로 열려 있는지 검사 |
| **Selection Sets (도면 요소 탐색)** | `(ssget "_X" '((8 . "A-WALL-INTR")))` | 지정한 전용 도면층 영역에 실제로 객체선들이 안전하게 분포되어 있는지 조사 |
| **DXF List (좌표 및 두께 확인)** | `(cdr (assoc 10 (entget ent)))` | 생성된 객체의 시작/끝점 또는 특정 정점 좌표가 수학 계산 좌표와 오차 범위 내인지 진단 |
| **Object Count (객체 개수 확인)** | `(sslength ss)` | 도면상에 중복선이나 이중 겹침 선이 발생하지 않고 필요한 수량만 깔끔하게 보존되었는지 검사 |

## 수행 수칙 (Rules)

1. **AutoCAD & BricsCAD 멀티 플랫폼 고려**: 호출하는 모든 드로잉 커맨드는 국제화 호환 접두사인 `._`로 묶어 처리해야 다른 언어 환경 캐드에서도 정상 작동합니다.
2. **독립적 상태 유지**: 하나의 테스트 검증이 끝날 때마다 사용했던 선택 세트(`setq ss nil`)나 기하 변수들을 메모리에서 비워내어 다른 검증 셀의 결과에 간섭하지 못하게 통제하십시오.
3. **환경 복원 여부 감사**: 어설션 검증이 다 끝난 후 사용자의 본래 캐드 제어 변수(`OSMODE`, `CMDECHO` 등)가 절대 흐트러짐 없이 유지되고 있는지 점검하십시오.
